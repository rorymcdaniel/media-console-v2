**Update AppBuilder.h:**

Add forward declaration at top (after existing forward declarations):
```cpp
class LocalPlaybackController;
```

Add member:
```cpp
std::unique_ptr<LocalPlaybackController> m_localPlaybackController;
```

**IMPORTANT:** Place m_localPlaybackController AFTER m_audioOutput and m_playbackState in the member declaration order. C++ destroys members in reverse declaration order. LocalPlaybackController's destructor stops its thread which may still reference m_audioOutput, so m_audioOutput must be destroyed AFTER m_localPlaybackController. Since reverse order means last-declared is first-destroyed, m_localPlaybackController should be declared AFTER m_audioOutput. This is already correct with the natural placement after the existing members.

**Update AppBuilder.cpp:**

Add include at top:
```cpp
#include "audio/LocalPlaybackController.h"
```

In build() method, after the receiver control section (after the existing "AppBuilder: receiver control initialized" log), add:

```cpp
    // Create audio playback controller
    m_localPlaybackController = std::make_unique<LocalPlaybackController>(
        m_audioOutput.get(), m_playbackState.get(), config.audio.deviceName, this);

    qCInfo(mediaApp) << "AppBuilder: audio playback controller initialized";
```

In the AppContext assignment section, add:
```cpp
    ctx.localPlaybackController = m_localPlaybackController.get();
```

The single LocalPlaybackController owning a single IAudioOutput enforces ALSA device exclusivity architecturally — there is no way for two sources to play simultaneously because only one controller exists.
